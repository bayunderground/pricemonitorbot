# План разработки MVP — PriceMonitorBot

Связан с CLAUDE.md — там зафиксированы технические ограничения и обоснования решений,
здесь — последовательность работ.

## Принятые решения перед стартом

- Тарифы: 3 уровня по лимиту товаров (Start/Business/Pro), сетка предложена в CLAUDE.md.
- Anti-detect для Ozon-парсера — с первого спринта, где он нужен. Ротация прокси — осознанно
  не в MVP, добавляется позже при необходимости.
- Сначала доводим до конца пайплайн на одном маркетплейсе (Wildberries — он проще), потом
  подключаем Ozon. Это снижает риск застрять на anti-bot защите Ozon в начале проекта и не
  иметь вообще ничего рабочего.

## Этап 0 — Инфраструктура и скелет проекта

**Задачи**
- Инициализация репозитория, структура каталогов по CLAUDE.md.
- `requirements.txt`, виртуальное окружение. Установка PostgreSQL нативно (без Docker) для локальной разработки.
- Настройка SQLAlchemy + Alembic, миграция с начальной схемой: `users`, `tracked_products`, `price_history`, `subscription_plans`, `subscriptions`, `payments`.
- `config.py` на pydantic-settings, `.env.example`.
- Базовое логирование (структурированные логи, уровень из env).

**Критерии готовности**
- `alembic upgrade head` отрабатывает без ошибок на чистой машине с установленным PostgreSQL.
- Таблицы созданы согласно схеме из CLAUDE.md.

## Этап 1 — Скелет Telegram-бота

**Задачи**
- Инициализация aiogram-бота, long polling для разработки.
- `/start`, `/help` — приветствие и краткая инструкция.
- Middleware: получение/создание пользователя в БД по `telegram_id` при первом обращении.

**Критерии готовности**
- Бот отвечает на `/start` и `/help` в тестовом чате.
- При первом сообщении нового пользователя в `users` появляется запись.

## Этап 2 — Парсер Wildberries

**Задачи**
- `ParserAdapter` — абстрактный интерфейс (`fetch(url_or_sku) -> ParseResult`).
- Реализация `WildberriesParser` через httpx к внутренним JSON-эндпоинтам карточки товара.
- Нормализация входа: пользователь может прислать как ссылку, так и голый артикул — оба варианта приводятся к одному виду.
- Обработка ошибок: товар не найден, эндпоинт недоступен/изменился, таймаут.
- Юнит-тесты на сохранённых JSON-фикстурах (без сети).

**Критерии готовности**
- На наборе из 10–15 реальных артикулов WB (ручной прогон, не в CI) парсер корректно возвращает цену и наличие.
- Юнит-тесты проходят в CI без сетевых вызовов.

## Этап 3 — Команды `/add`, `/list`, `/remove` (только WB)

**Задачи**
- `/add <ссылка или артикул>`: валидация, первый парсинг, сохранение в `tracked_products`.
- `/list`: список товаров пользователя с последней известной ценой/наличием.
- `/remove`: инлайн-клавиатура для выбора товара на удаление.
- Заглушка лимита товаров (например, дефолт 5 без реальной подписки) — чтобы не блокировать дальнейшую разработку до готовности платёжного модуля.

**Критерии готовности**
- Полный цикл добавления/просмотра/удаления товара WB через бота работает end-to-end на тестовой БД.

## Этап 4 — Планировщик и уведомления (WB)

**Задачи**
- APScheduler: `AsyncIOScheduler` + `SQLAlchemyJobStore` на той же Postgres.
- Job на регулярный обход всех активных `tracked_products` с `marketplace = wildberries`.
- Сравнение нового результата с `last_price`/`last_in_stock`; при изменении — запись в `price_history`, обновление `tracked_products`, отправка уведомления пользователю.
- Изоляция ошибок: сбой парсинга одного товара не прерывает обход остальных, ошибка логируется.
- Backoff при временных сетевых ошибках (retry с задержкой, не бесконечный).

**Критерии готовности**
- Ручное изменение цены у тестового товара в БД конкурента (или мок) приводит к уведомлению в Telegram при следующем цикле планировщика.
- Искусственная ошибка в одном товаре не останавливает обработку остальных в том же запуске job'а.

## Этап 5 — Парсер Ozon (Playwright + anti-detect, отдельный Docker-сервис)

**Задачи**
- `services/ozon-parser-service/`: лёгкий HTTP-сервис (FastAPI) на Playwright (headless Chromium) + playwright-stealth, упакованный в Dockerfile/docker-compose.yml. Это единственная часть проекта на Docker.
- Реалистичные паузы между действиями, рандомизация UA/viewport в разумных пределах.
- Детект капчи/блокировки → явная ошибка в ответе сервиса, **без попыток обхода**.
- Тесты на сохранённых HTML-снэпшотах страниц товара (без живых обращений в CI).

**Критерии готовности**
- Контейнер `ozon-parser-service` поднимается локально (`docker compose up -d --build`) и отвечает на HTTP-запрос с тестовым SKU.
- На наборе из 10–15 реальных SKU Ozon (ручной прогон) сервис корректно возвращает цену и наличие в большинстве случаев.
- При капче/блокировке сервис возвращает понятную ошибку, а не падает и не зависает.
- Юнит-тесты на фикстурах проходят в CI.

## Этап 6 — Интеграция Ozon в общий пайплайн

**Задачи**
- `parsers/ozon.py` в основном процессе — HTTP-клиент к `ozon-parser-service` (сам Playwright в основном процессе не запускается).
- `/add` распознаёт ссылки/артикулы Ozon, выбор адаптера по `marketplace`.
- Разная частота опроса по площадке (`OZON_PARSE_INTERVAL_HOURS` реже, чем `WB_PARSE_INTERVAL_HOURS`) — Ozon тяжелее по ресурсам и риску блокировок.
- Обновление `/list` и job'а планировщика для работы с обеими площадками.
- Обработка недоступности `ozon-parser-service` (контейнер не поднят/упал) — понятная ошибка в логах, а не падение всего job'а.

**Критерии готовности**
- Пользователь может добавлять и отслеживать товары обеих площадок одновременно, уведомления приходят корректно по каждой.
- Если контейнер `ozon-parser-service` недоступен, обработка WB-товаров продолжается без сбоев.

## Этап 7 — Тарифы и подписка через Telegram Stars

**Задачи**
- Сидинг `subscription_plans` тремя тарифами (Start/Business/Pro).
- `/subscribe`: инлайн-клавиатура выбора тарифа → `createInvoiceLink` с `currency=XTR`, `subscription_period=2592000`, `provider_token=""`.
- Обработчик `successful_payment`: создание/продление `subscriptions`, запись в `payments` (идемпотентно по `telegram_payment_charge_id`).
- `/status`: текущий тариф, лимит товаров, использовано/доступно, дата следующего списания.
- Реальная проверка лимита товаров по тарифу в `/add` (вместо заглушки из Этапа 3).
- При истечении подписки — приостановка уведомлений + сообщение с предложением продлить (а не удаление товаров).
- Перед реализацией — свериться с актуальной документацией Telegram Bot API по Payments ещё раз (раздел менялся за последний год).

**Критерии готовности**
- Тестовый платёж в Stars test-окружении Telegram проходит, подписка активируется на 30 дней.
- `/add` блокирует добавление товара сверх лимита тарифа с понятным сообщением.
- При истечении `current_period_end` без продления уведомления по товарам пользователя прекращаются, `/status` показывает предложение продлить.

## Этап 8 — Деплой на VPS

**Задачи**
- `docker-compose.prod.yml`: bot + postgres (+ при необходимости отдельный контейнер для Playwright-воркера, если ресурсы хоста потребуют разделения).
- Секреты — через `.env` на сервере, не в репозитории.
- Резервное копирование Postgres (минимум — cron c `pg_dump` на хосте или в отдельном контейнере).
- Базовый мониторинг падений (логи + хотя бы уведомление админу в Telegram при критической ошибке/падении планировщика).
- Restart-policy в Docker Compose, чтобы контейнеры поднимались сами после падения/перезагрузки сервера.

**Критерии готовности**
- Бот развёрнут и работает на VPS, переживает рестарт контейнера и перезагрузку сервера без потери данных.
- Резервная копия БД создаётся автоматически и проверена восстановлением хотя бы раз вручную.

## За рамками MVP (бэклог, не делать сейчас)

- Ротация прокси для парсера Ozon.
- Дополнительные площадки (Яндекс.Маркет и т.п.).
- Графики истории цены, расширенная аналитика для пользователя.
- Внешние платёжные шлюзы (YooKassa и др.) — явно исключены на этом этапе.
- Бесплатный trial-период подписки.
- Мультиязычность интерфейса бота.
